## 📖SOLID 원칙과 DRY
> 1. **SOLID 원칙** : 객체 지향 설계의 5가지 원칙
> 2. **DRY 원칙** : 코드의 재사용성과 유지보수성 높이기
---
### 1. SOLID 원칙 개요
객체지향 설계의 **변경에 강한 구조**를 만들기 위한 5가지 원칙

|원칙|명칭|핵심 개념|한 줄 요약|
|--|--|--|--|
|S|SRP (단일 책임)|클래스는 하나의 기능만 담당|수정해야 할 이유는 오직 하나여야 한다.|
|O|OCP (개방-폐쇄)|"확장에는 열림, 수정에는 닫힘"|기존 코드를 건드리지 않고 기능을 추가하라. (추상화/인터페이스 활용)|
|L|LSP (리스코프 치환)|자식은 부모의 역할을 완벽히 수행|부모 대신 자식을 넣어도 프로그램은 똑같이 돌아야 한다.|
|I|ISP (인터페이스 분리)|거대한 인터페이스보다 구체적인 여러 개|사용하지 않는 기능에 의존하도록 강제하지 마라.|
|D|DIP (의존 역전)|구체화가 아닌 추상화에 의존|상위 모듈은 하위 모듈의 구현에 휘둘리지 않아야 한다.|

> `SRP`에서 말하는 책임은 '역할'이 아니라 '변경의 이유(change reason)'다.
> 
> `OCP`는 '미래를 예측해 추상화하라'는 뜻이 아니라,
> '변경이 반복되는 지점을 기준으로 추상화하라'는 원칙이다.
> 
> `DIP`는 인터페이스를 '만드는 것'이 아니라,
> 의존성의 방향을 '제어 흐름과 반대로' 설계하는 것이다.

### 2. DRY (Don't Repeat Yourself)
#### ✅ 정의
- **"모든 지식은 시스템 내에서 단 한 번만, 명확하고 신뢰할 수 있게 표현되어야 한다."**
- 단순히 글자 형태의 중복을 피하는 것이 아니라, **비즈니스 로직(도메인 지식)의 중복**을 막는 것이 본질
  
    > `DRY`의 적은 중복이 아니라, '중복의 원인이 다름에도 불구하고 합쳐진 코드'다.

#### ⚠️ 주의사항 (WET 주의)
- **우발적 중복**: 코드 모양은 같아도 의도가 다른 코드를 억지로 합치면 결합도가 높아짐.
- **성급한 추상화**: 나중에 바뀔지도 모르는 걸 미리 공통화하면 수정이 더 힘들어질 수 있음.

#### ➕ 예시
```python
# ❌ 중복된 전처리 로직
def train_model():
    data = load_data()
    data = (data - mean) / std  # 정규화
    # 학습 코드...

def evaluate_model():
    data = load_data()
    data = (data - mean) / std  # 동일한 정규화 반복!
    # 평가 코드...

# ✅ DRY 적용
def normalize(data):
    return (data - mean) / std

def train_model():
    data = normalize(load_data())
    # 학습 코드...

def evaluate_model():
    data = normalize(load_data())
    # 평가 코드...
```

### 3. SW 개발 3대 원칙
1. **KISS** : Keep It Simple Stupid!
    - 디자인은 단순할수록 좋음. 불필요한 복잡성은 유지보수를 어렵게 만듦.
2. **YAGNI** : You Ain't Gonna Need It
   -  나중에 필요하겠지?"라는 생각으로 미리 만들지 말기.
3. **DRY** : Don't Repeat Yourself
   - 똑같은 일을 두 번 반복하지 말기

> 💡 **Tip** : 모든 원칙을 100% 엄격하게 지키려고 하기보다, 프로젝트의 규모와 팀의 상황에 맞춰 적절한 **트레이드 오프(Trade-off)**를 찾는 것이 중요!
